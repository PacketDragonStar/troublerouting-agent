# 基于 Microsoft AutoGen 的网络排障多智能体系统 落地方案

## 1. 项目目标与范围

构建一个**7×24小时自动化网络排障智能体团队**，能够：

- 接收自然语言故障报告
- 自动执行只读诊断命令（ping/traceroute/SNMP/日志拉取）
- 多智能体协作推理根因
- 生成修复建议并（在可控范围内）自动执行
- 输出标准化排障报告和操作记录
- 与企业现有工单、监控系统无缝集成

最终交付物是一套可容器化部署的**AutoGen多智能体服务**，通过API或Webhook接收故障，返回诊断结果与处理状态。

## 3. 智能体角色设计（六人团队）

### 3.1 Dispatcher（调度员）

- **职责**：接收用户报障或监控告警，解析关键信息（时间、设备、现象），生成结构化“故障摘要”，指派下一行动。
- **输入**：自然语言告警文本（如“总部三楼无线漫游异常”）。
- **输出**：结构化摘要 + 调查指令（如“请Investigator检查AC 10.0.0.1的CPU和关联AP状态”）。
- **提示词要点**：强调提取关键字段，不进行诊断，可引导补充信息。

### 3.2 Investigator（调查员）

- **职责**：唯一有权访问生产网络执行**只读命令**的Agent。根据指令逐步收集数据，每次返回结果并分析下一步。
- **工具集**（通过MCP挂载）：
  - `ping` / `traceroute`
  - `show interface / show log / show bgp summary`
  - `snmpget` (CPU、内存、端口错误计数)
  - `query_monitoring_api` (从Prometheus/Zabbix拉取时间序列)
- **输出**：结构化“症状数据报告”，包含关键指标和初步判断（如“CPU正常，但Gi0/1存在大量CRC错误”）。
- **安全限制**：MCP层强制黑名单/白名单，仅允许show/display类命令，禁止任何config模式。

### 3.3 Diagnostician（诊断专家）

- **职责**：综合分析所有症状、拓扑信息、历史案例，给出最可能的根因。
- **工具集**：
  - `search_case_library(query)`：检索向量化故障案例库（Chroma + text2vec）
  - `get_topology_context()`：查询实时拓扑（通过CMDB或LLDP采集）
- **输出**：根因分析报告，附带置信度。例如：“根因：上行光模块老化导致CRC错误，置信度92%”。
- **逻辑**：基于推理链，必要时可要求Investigator补充特定数据，但限制补查次数≤2次。

### 3.4 Solution Engineer（方案工程师）

- **职责**：根据诊断结果，制定可操作的修复方案，包括配置命令、验证步骤、风险评级。
- **工具集**：
  - `generate_remediation_plan()`：基于模板生成变更指令
  - `validate_config(commands)`：在虚拟数字孪生环境（如果存在）或通过语法检查器模拟
- **输出**：修复方案，如“建议更换光模块（低风险）或临时切换备用链路（中风险）”。

### 3.5 Safety Officer（安全/合规审计官）

- **职责**：独立审核修复方案，防止误操作、合规越界。拥有**一票否决权**。
- **工具集**：
  - `check_compliance(change)`：比对变更窗口策略、审计日志要求
  - `risk_analysis(impact)`：量化业务影响范围
- **输出**：审核结论（批准/有条件批准/驳回），明确操作前提（如“仅在凌晨3点且备份配置已完成条件下执行”）。

### 3.6 Reporter（报告员）

- **职责**：收集整个对话中的所有产出，生成标准化排障报告、操作记录，并推送至工单系统或通知渠道。
- **工具集**：
  - `create_ticket(summary)`：调用ServiceNow/Jira API
  - `send_notification(recipient)`：邮件/即时通讯通知
- **输出**：最终《故障处理报告》（Markdown/PDF），包含全程时间戳和所有Agent输出摘要。

> **为什么是六个人？** 这种分工对标真实NOC（网络运营中心）的层级：一线(Dispatcher/Investigator) → 二线(Diagnostician/Solution) → 变更委员会(Safety Officer) → 归档(Reporter)。每个角色职责清晰，便于后期单独训练或替换。

## 5. 工具集成与MCP实现

所有Agent不直接调用任何网络库，而是通过**MCP Server**暴露工具。这样做的好处：

- **安全隔离**：工具服务器运行在受限容器中，仅开放必要端口和命令白名单。
- **多Agent复用**：多个智能体共享同一套工具，无需各自配置。
- **可观测性**：所有工具调用自动生成审计日志。

MCP Server 提供以下 namespace：

- `network.readonly` : 所有show/ping命令
- `network.case` : 案例检索
- `system.ticket` : 工单操作
- `system.notify` : 通知

AutoGen通过`function_map`绑定MCP工具，或使用`autogen-ext[mcp]`扩展直接连接。

## 7. 部署方案

### 7.1 容器化交付

- 所有Agent运行在一个 **Kubernetes Pod** 中，共享AutoGen运行时。
- MCP Server作为Sidecar容器，注入网络访问权限（通过NetworkPolicy限制）。
- 使用Redis作为GroupChat的状态存储，避免单点丢失。

### 7.2 与企业系统对接

- **入口**：接收Webhook（监控系统/企业微信/邮件解析）
- **出口**：回写工单系统、发送即时消息、更新CMDB状态字段
- **配置**：网络设备清单、拓扑数据从CMDB API实时拉取，确保最新。

### 7.3 高可用

- 无状态设计，多副本部署，共享Redis实现群聊状态同步。
- LLM调用统一走网关，支持限流和故障切换（多API Key）。

## 9. 预期效益与风险评估

- **效益**：平均故障响应时间（MTTR）降低40%，减少夜间值班人力，知识沉淀在案例库中持续增强。
- **风险与对策**：
  - *Agent幻觉导致错误配置* → Safety Officer强制拦截，高风险操作必须人工二次确认。
  - *LLM调用延迟* → 使用缓存相似病例，关键命令本地执行。
  - *工具滥用* → MCP白名单+操作回放审计。
